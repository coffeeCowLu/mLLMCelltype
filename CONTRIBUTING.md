# Contributing to mLLMCelltype

Welcome to the mLLMCelltype community! We appreciate your interest in contributing to a multi-LLM consensus framework for cell type annotation.

## 🎯 Ways to Contribute

### 📝 **Code Contributions**
- **New LLM Provider Support**: Help integrate additional language models
- **Performance Optimizations**: Improve consensus algorithms and cost efficiency  
- **Bug Fixes**: Address issues reported by the community
- **Feature Enhancements**: Extend functionality based on user feedback

### 📚 **Documentation**
- **Tutorials**: Create guides for specific use cases
- **API Documentation**: Improve function documentation
- **Translations**: Help localize documentation to more languages
- **Examples**: Add real-world case studies and workflows

### 🧪 **Testing & Validation**
- **Benchmarking**: Test on new datasets and tissues
- **Cross-platform Testing**: Validate on different systems
- **Performance Testing**: Evaluate scalability and resource usage
- **Biological Validation**: Verify annotation accuracy

## 🚀 Getting Started

### 1. **Fork and Clone**
```bash
git clone https://github.com/your-username/mLLMCelltype.git
cd mLLMCelltype
```

### 2. **Development Setup**
```r
# Install development dependencies
devtools::install_dev_deps()

# Install package in development mode
devtools::load_all()
```

### 3. **Create Feature Branch**
```bash
git checkout -b feature/your-feature-name
```

## 📋 Development Guidelines

### **Code Standards**
- Follow existing code style and conventions
- Add comprehensive documentation for new functions
- Include unit tests for new functionality
- Ensure R CMD check passes without errors

### **Commit Message Format**
```
type(scope): brief description

More detailed description if needed

- List changes made
- Reference issue numbers if applicable
```

**Types**: feat, fix, docs, test, refactor, chore

### **Pull Request Process**
1. **Pre-submission Checklist**:
   - [ ] Code follows project conventions
   - [ ] All tests pass locally
   - [ ] Documentation updated
   - [ ] CHANGELOG updated if applicable

2. **Submit PR** with:
   - Clear title and description
   - Link to related issues
   - Summary of changes made
   - Testing performed

3. **Review Process**:
   - Maintainers will review within 48 hours
   - Address feedback promptly
   - Maintain civility and respect

## 🐛 Bug Reports

### **Before Reporting**
- Search existing issues for duplicates
- Test with latest version
- Prepare minimal reproducible example

### **Include in Report**
- mLLMCelltype version
- R/Python version and OS
- LLM models used
- Complete error messages
- Minimal code example

## 💡 Feature Requests

### **Good Feature Requests Include**:
